# Deploy Button + Server Management Design

How the Aeroform deploy button gets HTTPS onto a fresh VPS with zero user
configuration beyond the Linode OAuth tap, and how the user manages their
server from their phone.

## Problem

PocketBase has real auth, so the API is safe. But without TLS the credentials
fly in plaintext. Every public deployment needs HTTPS. Users don't have
domains, shouldn't need to create accounts with third parties, and shouldn't
touch a terminal.

## Decision: Caddy + sslip.io (zero-account automatic TLS)

### How it works

[sslip.io](https://sslip.io) is a free DNS service that maps
`<ip>.sslip.io` to `<ip>`. So `45.33.21.7.sslip.io` resolves to
`45.33.21.7`. No signup, no DNS config.

[Caddy](https://caddyserver.com) is a web server that auto-provisions
Let's Encrypt TLS certificates with zero config. Point it at a domain and
it handles ACME challenges, cert issuance, and renewal automatically.

Combined: cloud-init installs Caddy, writes a two-line Caddyfile using the
VPS IP, and Let's Encrypt issues a cert for `{ip}.sslip.io`. No accounts,
no domains, no interaction.

### Caddyfile (generated by cloud-init)

```
{IP}.sslip.io {
    reverse_proxy localhost:8090   # PocketBase
}
```

That's it. Caddy handles:
- TLS cert provisioning (Let's Encrypt ACME HTTP-01 challenge on port 80)
- Automatic cert renewal (every 60 days)
- HTTPS termination
- HTTP → HTTPS redirect

### Requirements

- Ports 80 and 443 open (default on fresh Linode/DO VPS)
- Public IPv4 address (standard on any VPS)
- No accounts, no tokens, no DNS configuration

## Caddy in Docker Compose vs System

Two options for where Caddy runs:

### Option A: System Caddy (recommended for deploy button)

Caddy runs as a systemd service on the host, outside Docker. Cloud-init
installs it via `apt install caddy`.

Pros:
- Survives `docker compose down`
- Can proxy to any local port
- Standard Linux admin pattern
- Cert storage in `/var/lib/caddy`

Cons:
- Not managed by docker compose
- Separate upgrade path

### Option B: Caddy in Docker Compose

Add Caddy as a service in docker-compose.yml.

Pros:
- Everything in one compose file
- Single `docker compose up` brings up everything

Cons:
- If compose goes down, TLS goes down
- Port binding conflicts if host already has something on 80/443
- Cert volume management

**Recommendation: Option A (system Caddy).** The TLS layer should be
independent of the application stack. If a user does
`docker compose down` to debug, they shouldn't lose HTTPS.

## Proxying Internal Services

Once Caddy is the front door, it can also proxy the knowledge stack
services that Flutter needs access to. But we DON'T do this — Flutter
still talks only to PocketBase, and PocketBase proxies internally via
the Go reverse proxy in `proxy.go`.

```
Flutter → Caddy (TLS) → PocketBase (:8090)
                              ├→ /api/pocketcoder/proxy/observability/* → SQLPage
                              ├→ /api/pocketcoder/proxy/knowledge/*    → OpenNotebook API
                              └→ /api/pocketcoder/proxy/memory/*       → poco-memory
```

Caddy's only job is TLS termination. Routing and auth stay in PocketBase.

## Alternative: Cloudflare Tunnel (power users / manual setup)

For users who want Cloudflare's network (DDoS protection, edge caching)
or who are behind CGNAT (no public IP):

```bash
# Requires free Cloudflare account + API token
cloudflared tunnel create pocketcoder
cloudflared tunnel route dns pocketcoder pc-{id}.cfargotunnel.com
```

This is NOT the default deploy button path because it requires a
Cloudflare account signup. Document as an alternative in the user guide.

Cloudflare Tunnel advantages:
- Works behind CGNAT / no public IP
- DDoS protection
- No ports to open
- Edge caching

Cloudflare Tunnel disadvantages:
- Requires account (blocks deploy button automation)
- Traffic routes through CF edge (sovereignty concern)
- Dependency on third-party service

## sslip.io Reliability

sslip.io is run by a single maintainer (Brian Cunnie) but:
- It's open source (can self-host if it goes down)
- nip.io is an identical alternative
- The DNS service is trivially simple — just returns the IP embedded in the name
- No data dependency — if it goes down, existing certs still work (they renew via ACME, not DNS)
- Fallback: user can point their own domain at the IP and update the Caddyfile

If sslip.io goes down permanently, a one-line change in cloud-init switches
to nip.io, sslip.cc, or a self-hosted equivalent.

## Decision: NixOS (game console model)

### Why NixOS over Ubuntu

The target user is someone with a phone, not a terminal. The server should
behave like a game console — an appliance with a few big buttons, not a
system to administer.

NixOS gives us:
- **Declarative config** — the entire server is a single `.nix` file, no drift
- **Atomic rollback** — bad update? Boot into the previous generation
- **Reproducible** — every deploy produces the exact same system
- **No "apt upgrade broke everything"** — generations are immutable

Ubuntu + cloud-init is a fragile sequence of shell commands that can fail
halfway through. NixOS describes the end state and gets there atomically.

Linode supports NixOS images. The bootstrap is: write `configuration.nix`,
run `nixos-rebuild switch`. Same one-tap deploy from Flutter, just a
different (more reliable) bootstrap mechanism.

### NixOS configuration (sketch)

```nix
{ config, pkgs, ... }: {
  # Caddy — auto-TLS via sslip.io
  services.caddy = {
    enable = true;
    virtualHosts."${ip}.sslip.io".extraConfig = ''
      reverse_proxy localhost:8090
    '';
  };

  # Docker for the PocketCoder stack
  virtualisation.docker.enable = true;

  # Firewall
  networking.firewall.allowedTCPPorts = [ 80 443 ];

  # Auto-start PocketCoder on boot
  systemd.services.pocketcoder = {
    after = [ "docker.service" ];
    wantedBy = [ "multi-user.target" ];
    serviceConfig = {
      WorkingDirectory = "/opt/pocketcoder";
      ExecStart = "${pkgs.docker-compose}/bin/docker-compose up";
      ExecStop = "${pkgs.docker-compose}/bin/docker-compose down";
      Restart = "always";
    };
  };
}
```

### The three buttons (game console ops)

The user's phone shows three recovery actions. No terminal needed.

Uses gaming mental model — everyone already understands this.

```
┌─────────────────────────────────┐
│  SERVER CONTROLS                │
│                                 │
│  [ RESTART GAME ]               │
│    Restarts PocketCoder.        │
│    Your data is safe.           │
│                                 │
│  [ RESTART CONSOLE ]            │
│    Reboots the entire server.   │
│    Takes about a minute.        │
│                                 │
│  [ LOAD SAVE ]                  │
│    Restore from a backup.       │
│                                 │
└─────────────────────────────────┘
```

#### 1. Restart App (restart docker stack)

Restarts the PocketCoder docker compose stack. Fixes app-level issues
(hung process, bad config, OOM).

Implementation: PocketBase already has the Docker Socket Proxy pattern.
Flutter calls a PB endpoint → PB triggers `docker compose down && up`
via the socket proxy. No SSH, no Linode API.

Caddy stays up (system service) — server remains reachable over HTTPS,
returns 502 briefly during restart.

#### 2. Restart Server (reboot NixOS)

Full system reboot. Fixes system-level issues (kernel panic, resource
exhaustion, stuck systemd units).

Implementation: Linode API — `POST /v4/linode/instances/{id}/reboot`.
Flutter already has the Linode OAuth token from deployment. No SSH needed.

NixOS boots into the same declarative config. Docker starts via systemd.
Everything comes back to the same known-good state.

#### 3. Backup / Restore

Two levels of backup:

**Linode snapshots (full disk):**
- Create: `POST /v4/linode/instances/{id}/backups`
- Restore: `POST /v4/linode/instances/{id}/backups/{backupId}/restore`
- Covers everything — DB, config, Docker volumes
- Linode API, no SSH needed

**PocketBase data export (fine-grained):**
- Export: PB endpoint that dumps the SQLite DB
- Download to phone or upload to object storage
- Lighter weight, can be scheduled (cron job)
- Useful for migrating between servers

### Why this works for normies

| Problem | Game console fix |
|---|---|
| App is slow/broken | Restart App button |
| Server is unresponsive | Restart Server button |
| I broke my data | Restore from backup |
| System update broke things | NixOS rolls back automatically |
| I want to move servers | Export data, deploy new, import |

No SSH. No terminal. No googling error messages. Three buttons.

| Mode | Target user | Monthly cost | Auth flow | Management |
|---|---|---|---|---|
| **Linode** | Default / easy | ~$12/mo | OAuth (one-tap) | Self-hosted, game console buttons |
| **Elestio** | Normie / hands-off | ~$18/mo | Elestio account + API token | Fully managed by Elestio |

All three use the same PocketCoder docker-compose stack. The only
differences are the API calls to create the server and the auth flow.

### IAP model (pay per deploy)

The deploy button is an in-app purchase. Each purchase unlocks a
**24-hour deploy window**.

- User pays IAP → can deploy for 24 hours
- Deploy fails or needs retry? Still in the window, no extra charge
- Want a second server? New IAP purchase
- Destroy and redeploy months later? New IAP purchase

Each deploy is a purchase. Simple, fair, no subscriptions from us.

### Revenue streams

| Deploy path | IAP | Recurring revenue | Notes |
|---|---|---|---|
| **Linode** | Yes | $10 one-time affiliate cash | Linode affiliate program pays $10 per new paying customer |
| **Elestio** | Yes | ~$3.60/mo (20% rev share) | Elestio OSS author program, recurring as long as user runs the service |
| **Hetzner** | Yes | $10 one-time credit | Hetzner referral, credit only (not cash) |

**Elestio is the best recurring revenue play.** 20% of ~$18/mo = ~$3.60/mo
per user, indefinitely. Linode affiliate is a one-time $10 cash payment.
Hetzner is a one-time $10 credit (not even cash).

**Revenue streams cannot be stacked.** When a user deploys via Elestio and
picks Linode as the underlying provider, Elestio provisions the server on
Elestio's own Linode account. The user never signs up for Linode directly,
so no referral bonus flows to PocketCoder. You earn either the Elestio
rev share OR the cloud provider referral, never both.

Elestio has a catalog of 400+ open source projects. Getting listed means:

### Provider auth differences

**Linode (OAuth — best UX):**
Flutter opens browser → user logs into Linode → OAuth token returned →
app creates server automatically. True one-tap.

**Elestio / Hetzner (API token paste):**
User must create an account on the provider's website, generate an API
token manually, and paste it into the app. More friction, but acceptable
for their target users (Elestio users want managed; Hetzner users are
power users who don't mind).

The app should provide clear step-by-step instructions with screenshots
for the token paste flow.

### Transparency

Show users all three options with honest pricing and trade-offs.
PocketCoder is an open source sovereignty tool — transparency builds
trust. Don't hide the cheap option to push users toward the revenue-
generating option.

## Cloud-Init / Bootstrap (NixOS)

### Linode NixOS bootstrap path

Linode does NOT offer a native NixOS image. Three options:

**Option A: Custom image upload (recommended)**
Build a NixOS image with `nixos-generators` (has a `linode` format),
upload once to Linode as a custom image, stamp out instances from it.
Deterministic, reproducible, fast boot. Image must be raw ext4 + gzip,
under 5GB compressed.

**Option B: nixos-anywhere on Ubuntu base**
Boot Ubuntu, run `nixos-anywhere` via SSH to kexec into NixOS installer
and convert in-place. More flexible (can customize per-deploy) but
requires a two-step process and SSH access post-boot.

**Option C: Ubuntu + cloud-init (ship first, NixOS later)**
Skip NixOS for v1. Boot Ubuntu, cloud-init installs Docker + Caddy +
PocketCoder. Works today with zero NixOS complexity. Upgrade to NixOS
in a future release.

**Decision:** Ship with Option A (custom NixOS image). The image build
is a one-time step. Every deploy after that is identical to Ubuntu —
the user taps deploy and gets a server. But the server is an appliance
with atomic rollback, no drift, and the game console guarantees from
day one. No reason to ship something worse and migrate later.

### Hetzner bootstrap

Same cloud-init approach. Hetzner has full cloud-init support with
`user_data` in the server creation API. Ubuntu base works identically.

### Elestio bootstrap

No bootstrap needed. Elestio handles everything. PocketCoder provides
an `elestio.yml` that tells Elestio how to deploy the docker-compose
stack. Elestio provisions the server, installs the stack, configures
SSL, and manages it going forward.

## Open Questions

1. **NixOS image build pipeline?** Need to set up `nixos-generators`
   with the `linode` format, define the base `configuration.nix`, build
   the image, and upload to Linode as a custom image. One-time setup,
   then every deploy stamps from the same image.

2. **Elestio partnership status?** Need to re-contact Elestio with the
   repo link and a clear pitch (not "hackathon project"). Lead with:
   AGPLv3, docker-compose stack, Linode deploy integration already built,
   looking for catalog listing + revenue share.

3. **IAP pricing?** Need to decide the deploy button price. Should be
   low enough to not deter users but high enough to be worth it.
   $2.99? $4.99? Research comparable IAPs.

4. **Rate limits?** Let's Encrypt allows 50 certs per registered domain
   per week. sslip.io is a single registered domain, so all sslip.io
   users share this limit. In practice this hasn't been an issue — LE
   has specific carve-outs for public suffix domains. If it becomes one,
   switch to nip.io or self-host.

5. **NixOS auto-updates?** NixOS supports unattended upgrades with
   automatic rollback if the health check fails. Could enable this so
   the server stays patched without user intervention — true appliance
   behavior. Relevant once NixOS path is implemented.
